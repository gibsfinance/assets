## Session: Knex ‚Üí Drizzle Phase 4 (2026-03-25)

### Phase 4: Complex Query-Composition Functions (completed)
Converted the 7 remaining Knex query-builder pattern functions to Drizzle ORM:
- `insertOrder` ‚Äî Knex transaction ‚Üí Drizzle transaction with DrizzleTx passthrough
- `getTokensUnderListId` ‚Äî Knex chainable ‚Üí Drizzle `$dynamic()` with typed select
- `getLists` ‚Äî Knex chainable with string-based where ‚Üí async Drizzle with `eq()`/`and()`
- `addHeaderUriExtension` ‚Äî Knex.QueryBuilder param ‚Üí generic `<T extends $dynamic()>`
- `addBridgeExtensions` ‚Äî Knex joins with aliased tables ‚Üí Drizzle `alias()` from pg-core
- `getListOrderId` ‚Äî Knex `.where().first()` ‚Üí Drizzle `eq()`/`ilike()`/`limit(1)`
- `applyOrder` ‚Äî The hardest: Knex `denseRank()/partitionBy()/CTE` ‚Üí raw `dsql`` template with window function

Updated all server-side callers:
- `db/sync-order.ts` ‚Äî Drizzle transaction + `isNotNull()`/`eq()` queries
- `server/list/utils.ts` ‚Äî `respondWithList` uses `eq()`/`asc()` on `$dynamic()` query
- `server/list/handlers.ts` ‚Äî All 5 handlers (merged, versioned, providerKeyed, all, tokensByChain) converted
- `server/image/handlers.ts` ‚Äî `getListTokens`, `getNetworkIcon`, `getImageByHash` all use Drizzle
- `collect/pumptires.ts` ‚Äî `getTokensUnderListId()` callers use `eq()`/`desc()`

Key decisions:
- `applyOrder` uses raw SQL (`dsql` template) because Drizzle `$dynamic()` cannot add SELECT columns after creation (needed for dense_rank window function)
- Two FROM clause variants in `applyOrder`: 'listToken' (for merged handler) and 'provider' (for image handler) to match the original Knex query patterns
- `addBridgeExtensions`/`addHeaderUriExtension` return type cast through `unknown` since adding joins changes Drizzle's type signature

Results: 8 fewer TS errors (52 ‚Üí 44), zero new errors. All 142 server tests passing.
